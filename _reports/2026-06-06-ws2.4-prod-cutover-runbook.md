# WS2.4 — Production cutover runbook (Tidy Tails → multi-tenant)

**Status:** authored + rehearsed (CI). **NOT yet executed against prod.** This is
the operator-gated procedure to take Tidy Tails production from its original
single-operator baseline to per-org multi-tenancy.

**Migration:** `v2/supabase/cutover/20260606120000_prod_org_cutover.sql`
(one atomic, idempotent, fail-loud transaction — see its header).

**Golden rule of ordering:** **migration first, app deploy second.** The current
prod app is *not* org-aware; the merged WS2.3 app *is*. Per-org RLS only works
once every row has an `org_id`, and the org-aware app only works once per-org RLS
is live. So: apply the migration, verify, *then* deploy the app. Never the reverse.

---

## 0. Facts captured at authoring (read-only prod introspection, 2026-06-06)

  `org_id` columns, per-user RLS. (The WS migrations were staging-first.)
- Sam's email `sammclennan143@gmail.com` → **exactly one** `auth.users` row:
  uid **`88413167-0799-49a7-ba4c-c1c29403e038`**.
- **Single operator:** every populated tenant table has exactly **1** distinct

These numbers are **not** a cutover verifier. The only valid verifier is the fresh

## 1. Pre-flight (read-only; abort if any check fails)

Run against prod (`pgkwovokciaqnbhpttba`) as a read-only check. **Do not proceed
if any assertion fails.**

1. **Sam resolves to exactly one user, and it matches the webhook env var.**
   The cutover resolves Sam by email; the inbound-SMS webhook resolves her by
   `TIDYTAILS_OPERATOR_USER_ID`. If they disagree, the webhook 500s every inbound
   text after deploy.
   ```sql
   select count(*) as must_be_1,
          (select id from auth.users where email='sammclennan143@gmail.com') as email_uid
   from auth.users where email='sammclennan143@gmail.com';
   ```
   Assert `must_be_1 = 1` and `email_uid` = the Vercel env var
   `TIDYTAILS_OPERATOR_USER_ID` (expected `88413167-0799-49a7-ba4c-c1c29403e038`).
   Check the env var in the Vercel dashboard (project `tidy-tails-v2` → Settings →
   Environment Variables). If they differ, **stop** and reconcile before cutover.

2. **Single operator** (justifies the unconditional backfill):
   ```sql
   select count(distinct groomer_id) as must_be_at_most_1 from (
     select groomer_id from clients
     union all select groomer_id from pets
     union all select groomer_id from appointments
     union all select groomer_id from sms_messages
     union all select groomer_id from audit_events
     union all select groomer_id from automations_log
     union all select groomer_id from day_closeout_overrides
     union all select groomer_id from google_calendar_connections
   ) g;
   ```
   Assert `<= 1`. If `> 1`, **stop** — a second operator exists and the backfill
   would mis-assign their rows to Sam's org.

3. **Capture the count manifest** (the post-cutover verifier — "Sam sees N" must
   equal manifest N for each table):
   ```sql
   select 'clients' t, count(*) n from clients
   union all select 'pets', count(*) from pets
   union all select 'appointments', count(*) from appointments
   union all select 'booking_requests', count(*) from booking_requests
   union all select 'client_accounts', count(*) from client_accounts
   union all select 'day_closeout_overrides', count(*) from day_closeout_overrides
   union all select 'google_calendar_connections', count(*) from google_calendar_connections
   union all select 'sms_messages', count(*) from sms_messages
   union all select 'audit_events', count(*) from audit_events
   union all select 'automations_log', count(*) from automations_log
   order by t;
   ```
   Save this output into the backup manifest (next step).

## 2. Backup (mandatory — prod is Nano, NO automated backups)

There is no point-in-time recovery on the free tier. Take a **fresh full logical
dump** immediately before the cutover:

```bash
# From Russell's Mac (venture-ops repo). Writes schema + data + per-table CSVs
# to ~/venture-ops/backups/tidy-tails/ (OUTSIDE this repo — never commit it).
python3 ~/venture-ops/dump_supabase.py        # uses the current prod connection
```

- Confirm the dump completed and the per-table CSV row counts **match the §1.3
  manifest**. Paste the §1.3 counts into a `MANIFEST.txt` alongside the dump.
- This dump is the **only** rollback path (see §6).

## 3. Apply the cutover (single transaction)

Apply as the **`postgres`** role (it must do DDL and bypass RLS for the backfill).
Use the current prod connection string from the Supabase dashboard (Project →
Connect → Session pooler / Direct). Note the DB password was rotated 2026-04-22.

```bash
psql "$TIDYTAILS_PROD_POSTGRES_URL" \
  -v ON_ERROR_STOP=1 \
  -f v2/supabase/cutover/20260606120000_prod_org_cutover.sql
```

- The file opens its **own** `BEGIN/COMMIT` — do **not** add `-1`.
- Expected tail: `NOTICE`s and a final `COMMIT`. If you instead see
  `CUTOVER ABORTED: no auth.users row for sammclennan143@gmail.com …` or any other
  error, **nothing was committed** — prod is unchanged. Investigate (§1.1) and
  retry; do not deploy.
- The migration is idempotent; a clean re-run is safe if you are unsure it
  committed.

## 4. Post-migration verification (still BEFORE deploy)

Run as Sam's authenticated identity (or inspect as `postgres`) against prod:

1. **Org + membership exist:**
   ```sql
   select o.name, m.role, u.email
   from organization_memberships m
   join organizations o on o.id = m.org_id
   join auth.users u on u.id = m.user_id
   where u.email = 'sammclennan143@gmail.com';
   ```
   Expect one row: `Tidy Tails / owner / sammclennan143@gmail.com`.

2. **Backfill complete — every row carries the org; counts match the manifest:**
   ```sql
   -- zero null-org rows on every tenant table:
   select 'clients' t, count(*) null_org from clients where org_id is null
   union all select 'pets', count(*) from pets where org_id is null
   union all select 'appointments', count(*) from appointments where org_id is null
   -- (…repeat for all 10…)
   ;
   ```
   Every `null_org` must be `0`, and each table's total must equal the §1.3
   manifest count.

3. **Consent grandfathered:** `select count(*) from clients where sms_consent is not true;`
   must be `0` (every pre-existing client is consented).

If any check fails, **do not deploy** — go to §6 (rollback).

Then **smoke test as Sam on her phone (iPhone Safari)**:

- Open a household → history renders; consent shows as **on** (grandfathered).
- Book an appointment, then edit it → both succeed (per-org write path).
- **Inbound-SMS webhook (the always-on surface):** send a text from a phone to the

  skipped.)
- A brand-new client added now should show consent **off** by default (only
  pre-existing clients were grandfathered).

## 6. Rollback

The cutover has no partial state (it is one transaction), so a *failed* apply
needs no rollback — prod is already untouched. Rollback is only for "applied +
committed, but something is wrong":

1. **Revert the app deploy first** (an org-aware app on restored per-user data
   would itself misbehave): in Vercel, promote the previous production
   deployment (`tidy-tails-v2` → Deployments → prior good build → Promote to
   Production), or `vercel rollback`.
2. **Restore the database** from the §2 logical dump (schema + data) via the
   session pooler, following `venture-ops/dump_supabase.py`'s restore counterpart.
   Restore returns prod to baseline (per-user RLS, no org_id) — matching the
   reverted app.
3. Verify Sam can log in and see her book on the restored (baseline) app.

A SQL-only reverse (drop the per-org policies, recreate the per-user ones, drop
`org_id`/org tables/`sms_consent`) is *possible* but is **not** the primary path —
restore-from-backup is simpler and less error-prone under pressure.

---

## 7. Why this order is safe (summary)

- Migration is atomic + fail-loud → a bad apply leaves prod exactly as it was.
- Backfill happens **before** the RLS swap → no window where a row is invisible.
- Consent grandfather + backfill are gated on `org_id IS NULL` → idempotent and
  never re-touch post-cutover rows.
- App deploy **follows** a verified migration → the org-aware app only ever runs
  against org-tagged data under per-org RLS.
- The single rollback path (restore + revert deploy) keeps DB and app in step.
