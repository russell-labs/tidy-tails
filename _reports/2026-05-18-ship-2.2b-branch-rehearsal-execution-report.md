## Migration mechanics

Both migrations were applied as **one named Supabase migration each** via
`apply_migration`, which wraps the whole body in a single `BEGIN`/`COMMIT` — so
each landed atomically. After each apply, `NOTIFY pgrst, 'reload schema'` was
issued so PostgREST's schema cache reflected the DDL before the REST checks ran.

- **Forward** = parent plan §4, Steps A→B→C: add `groomer_id` to the 6 in-scope
  tables, backfill to the rehearsal UID, `SET NOT NULL`; drop the 20 permissive
  policies and create 24 `auth.uid()`-scoped `groomer_*` policies; drop the
  `client_overview` view.
- **Rollback** = parent plan §5, Steps C⁻¹→B⁻¹→A⁻¹: recreate `client_overview`;
  drop the `groomer_*` policies and restore the 20 permissive policies; drop the
  `groomer_id` column.

`sam_review_responses` was out of scope and left untouched by both migrations.

## Remaining risks for the production cutover

1. **UID substitution is a hard gate.** The forward SQL artifact embeds the
   rehearsal UID `a49e3c18-…`. The production cutover **must** substitute
   Samantha's verified production `auth.users` UID. The branch's `auth.users` is
   isolated — Samantha's production UID does not exist on the branch, and the
   rehearsal UID does not exist in production.

2. **Samantha's production `auth.users` account must exist before cutover.**
   `groomer_id NOT NULL FK → auth.users` cannot be backfilled without it (parent
   plan §8 prerequisite 1). Not verified by this rehearsal — out of scope.

3. **The branch is a high-fidelity but not byte-identical copy of production.**
   Its schema was reconstructed from read-only introspection, not a `pg_dump` —
   column ordering may differ cosmetically. The rehearsal proves the migration
   *logic* and RLS *behavior*; it does not replace the fresh, verified logical
   backup that must be taken immediately before the cutover (parent plan §6).

4. **PostgREST schema-cache reload is a required runbook step.** After the
   forward migration, REST calls only saw the DDL change once
   `NOTIFY pgrst, 'reload schema'` was issued. The production cutover runbook
   must include this immediately after the migration.

5. **The v2 app was not exercised.** Phase 3 tested the authenticated *read
   path* via a direct authenticated REST call (the identical RLS surface) — not
   the Next.js app. The app-layer checks in parent plan §7 "Post-migration —
   app" (login page, `proxy.ts` redirect, session refresh, page rendering)
   remain a separate verification, run when v2 is deployed. The database-side
   R-1 closure is fully proven; app integration is not.

6. **DELETE policies are newly introduced.** The forward migration creates a
   `groomer_delete` policy on all 6 tables; v1 had no DELETE policy after
   2026-04-22. This is the deliberate parent-plan §4 Step B change. v1 HTML
   calls no `.delete()`, so v1 is unaffected — flagged so it is a conscious
   change at cutover, not a surprise.

7. **Anon write rejection manifests two ways.** Anon INSERT is hard-rejected
   (HTTP 401, RLS `WITH CHECK` violation). Anon UPDATE/DELETE return HTTP 200
   with 0 rows affected (the `USING` clause matches nothing for `anon`). Both
   mean "anon cannot write," but a cutover verifier checking only for a 4xx
   status would mis-read the UPDATE/DELETE case — verify by **row count**, not
   status code alone.

8. **The branch is billable and should be deleted.** The branch accrues
   ~$0.01344/hr while it exists. Per the approval it was kept alive until this
   report was complete — it now is. Deleting the branch and the gitignored
   `venture-ops/.env.rehearsal` file is the clean-up step (a separate action;
   not performed by this report). The two SQL artifacts and this report are
   durable; the branch is disposable.
