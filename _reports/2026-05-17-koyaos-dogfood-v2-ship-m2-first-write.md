Earlier sessions produced REQ-01 through REQ-27. Building M2 — the first
surface that writes to the operator's live database — surfaced one more.

---

## REQ-28 — Dry-run mode for a venture's first write surface

**What happened:** M2 is the first Tidy Tails surface that writes to Samantha's
live database (creating an appointment). The whole planning corpus gates live
writes behind an RLS-hardening migration (Ship 2.2b) — which is itself
hard-gated on prerequisites that are not close (an `auth.users` account, a
written backup procedure, a rehearsed rollback). Blocking the entire product
wedge on that infra ship was not acceptable; shipping a live write before it
was also not acceptable.

**Pattern that worked:** M2 was built with a **dry-run mode**. The write server
action runs the *complete* flow — auth re-check, input validation, pet/client
ownership check, insert-payload construction — and then, instead of persisting,
returns a result: a "demo" outcome in fixture mode, a "gated" refusal in live
mode (the write gate stays closed until 2.2b). Nothing is written. This let the
entire booking flow — form, review/confirmation, error states, the
ownership-rejection path — be built, browser-tested, and proven *before* a
single live row exists and *before* the security gate is lifted.

**Why it matters:** a first-write surface tangles two risks — (1) the write
*flow* might be wrong (weak validation, a missing ownership check, a malformed
payload), and (2) the *security posture* for writes might not be ready. Teams
routinely block the whole surface on (2), which leaves (1) unverified until the
gate lifts — and then the first real write is also the first test of the flow.
The dry-run decouples them: the flow is fully exercised while the gate stays
closed, so lifting the gate later is a small, late, low-risk switch rather than
a leap into the unknown.

**What Koya needs:** codify a **dry-run / write-rehearsal mode** as the
recommended pattern for any venture's first write surface — or any first write
to a not-yet-hardened store. The write action runs the full flow minus the
final persist, gated by the same fixtures/live flag that already governs reads
(REQ-12). Two hard requirements:

1. **The result state must be visually unmistakable as a non-write** — "nothing
   was saved", warning-styled, never a success affordance — so an operator can
   never mistake a rehearsal for a real save.
2. **Lifting the gate is one explicit, logged decision** — a single flag flip
   recorded in `docs/DECISIONS.md`, not an ambient config drift.

Koya could scaffold this: when it detects a surface's first write action,
generate it with a persist-gated dry-run path by default.

**Relation to prior REQs:** REQ-12 is the fixtures/live *read* dispatch pattern;
REQ-27 guards a deferred read path against schema drift; REQ-28 is the *write*
counterpart — rehearse the write flow fully before the security gate lifts.
