# KoyaOS dogfood lesson — Tidy Tails v2 Reminder Prep (fourth surface)

Earlier sessions produced REQ-01 through REQ-30. Building Reminder Prep — the
venture's fourth dry-run/gated surface, and the first whose action's real
effect reaches *outside the system and cannot be undone* — surfaced one more.

---

## REQ-31 — An "active" surface flag must never imply "automatic"; classify surfaces by side-effect reversibility

**What happened:** Reminder Prep is Tidy Tails v2's fourth dry-run/gated
surface. M2 (REQ-28), Log Groom (REQ-29), and Add Household (REQ-30) all write
**internal database rows** — a wrong row can be corrected or deleted. Reminder
Prep is the first surface whose action's real effect is an **irreversible
external side-effect**: an SMS delivered to a real person's phone. A sent text
cannot be unsent. The hard product rule for this surface is explicit and
permanent: the app *prepares* a reminder; Sam reviews and explicitly confirms
every individual SMS in-app; no automatic, no batch, no background sending —
and that rule holds **after** the cutover, not only before it.

**Pattern that worked:** model the prepared reminder as **inert data**. The
pure layer produces a `ReminderDraft` that carries `requiresExplicitConfirmation:
true` as a structural invariant, has no "sent" state, and the module exposes no
function that sends. The dispatch decision is never made in the pure or
preparation layer — it is a separate, per-event, human-confirmed step.

**Why it matters:** REQ-29 established that each write surface gets its own
flag, lifted by its own logged decision. The implicit lifecycle there was:
flag off → surface inert; flag on → surface active → operator confirms once →
write happens. For a database-row write that lifecycle is fine. For an
irreversible external dispatch it is **not** — because "active" is one short
step from "the platform may now fire this on its own." A reminder surface, even
with its send flag fully on, must still require a *fresh human confirmation for
every individual message*. The flag authorizes the **capability**; it never
authorizes an automatic, scheduled, or batch send. If KoyaOS models a surface's
flag as a plain active/inactive bit and lets "active" surfaces be driven by
schedulers or automations, the first irreversible-side-effect surface to go
active becomes an auto-texting machine — precisely what this venture's hard
rule forbids.

**What Koya needs:** add a side-effect dimension to the surface model.

1. Classify each write/send surface by **side-effect reversibility** —
   internal-DB-row (correctable) vs irreversible-external (SMS, email, payment,
   webhook). The per-surface flag (REQ-29) carries this classification.
2. For an irreversible-external surface, the platform **enforces per-event
   human confirmation** as a property of the surface, independent of flag
   state: "flag active" means *may send when a human confirms this specific
   event* — never an automatic, scheduled, or batch send.
3. No automation or scheduler may be wired to an irreversible-external surface.
   Its verifier confirms the only path to dispatch is a fresh, logged,
   single-event human confirmation.

**Relation to prior REQs:** REQ-28 is the dry-run pattern for the first write
surface; REQ-29 made per-surface flags independent of the global security gate;
REQ-30 flagged that dependent multi-row writes need a transactional wrapper.
REQ-31 is the side-effect dimension: a surface flag being "active" must never
collapse into "automatic" — an irreversible external dispatch stays human-
confirmed per event, permanently.
